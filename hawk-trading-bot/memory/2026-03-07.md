## DSL v5.2 Upgrades — edit_position discovery (18:35 UTC)

Jason pointed out edit_position supports targetMargin for adding/reducing size, and takeProfit for native TP on Hyperliquid.

### New capabilities integrated:

1. **Partial TP via targetMargin** (replaces counter-direction order hack)
   - `edit_position(targetMargin=<lower amount>)` → POSITION_REDUCED
   - Cleaner, no edge cases with leverage math on counter orders

2. **Native TP on Hyperliquid** via `edit_position(takeProfit: {price, orderType: "LIMIT"})`
   - Set at next tier's trigger price
   - HL executes instantly when price hits — no 3-min cron delay
   - CONFIRMED WORKING: SHORT TP at $30.362 fired, closed position at ~8% ROE

3. **Pyramiding — add to winners** via `edit_position(targetMargin=<higher amount>)`
   - Only in Phase 2 (confirmed trend), on tier upgrades
   - Adds 15% margin per pyramid, max 2 adds per trade
   - Entry price blends automatically from currentState response
   - ALO execution for fee savings

### Key learnings:
- `edit_position` is the Swiss Army knife: leverage, SL, TP, add, reduce — all in one
- `actionsPerformed` array confirms what actually happened
- `currentState` in response gives new size/entry after edit
- audit_get_strategy_history only shows strategy-level actions, not individual fills
- Native TP > DSL-managed TP for speed (instant vs 3-min cron)

## Bug: DSL archives TP fills as "external" (18:40 UTC)

When DSL sets a native TP via edit_position(takeProfit), and HL executes it, DSL's next run sees
no position in clearinghouse. Current logic: "position gone but no SL fill detected → archive as
external." It should also check if the TP order filled before defaulting to "external."

**Fix needed in dsl-v52.py:** Before archiving as "external", check:
1. If state has `lastSyncedTpPrice` set → check if a TP order was filled
2. If TP filled → archive as `archived_tp` (not `archived_external`)
3. Notify scanner with was_loss=False (it was a profit take)

This is important because "external" implies something outside DSL closed it, when actually
DSL's own native TP did the job. Correct attribution matters for trade logging.

## Bug fix: compute_candle_momentum was broken (19:55 UTC)

`compute_candle_momentum(candles, n_bars=1)` compared `completed[-1]` to `completed[-1]` — 
same candle, always 0%. This meant `mom_5m` and `mom_15m` were null for every scanner run,
so the scanner was falling back to polled price history (which had fewer data points).

**Fix:** Changed index to `completed[-(n_bars + 1)]` so n_bars=1 compares [-2] to [-1].

Also lowered volume_spike_ratio from 1.5 to 1.2 — was blocking valid signals (1.43x blocked
a 1.13% 15m move).
